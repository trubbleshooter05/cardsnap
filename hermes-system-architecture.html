Component Details

Scheduler (launchd)

System timer / cron

Triggers the Hermes gateway at scheduled times based on cron expressions defined in jobs.json.

Role: Time-based trigger source

Hermes Gateway

~/.hermes/hermes_run.py

Main orchestrator. Reads jobs.json and state.db, loads environment config from .env, executes job scripts, captures output, and routes results to Telegram/Obsidian.

Role: Central coordinator

jobs.json

~/.hermes/cron/jobs.json

Configuration source. Defines all scheduled jobs with cron expressions, skills to run, delivery channels, and metadata. Must be valid JSON with correct cron syntax.

Role: Job definitions

state.db

~/.hermes/state.db

Scheduler cache. SQLite database holding Hermes' internal scheduler state: parsed job objects, next run times, last run times, execution history. If corrupted or stale, Hermes fails to parse valid job schedules.

Role: Cached scheduler state

~/.hermes/.env

Environment configuration

API keys, credentials, and runtime settings for external services (OpenAI, Telegram, etc.). Loaded by Hermes gateway before skill execution.

Role: Credentials & secrets

Skill Script

e.g., ~/.hermes/skills/daily-sync/

Executable Python/shell script that performs the actual work. Invoked by hermes_run.py with system prompt and context. Outputs to stdout, which Hermes captures.

Role: Work executor

Telegram

Telegram Bot API

Receives execution results from Hermes gateway and sends messages to configured chat. Configured via job.deliver field and .env credentials.

Role: Real-time notification

Obsidian Logs

~/ObsidianVault/logs/

Daily automation logs and job execution records. Hermes appends run results to automation_runs.md for audit trail and debugging.

Role: Persistent record
Execution Flow

What Happens When a Job Runs

1
Scheduler fires

System timer (launchd) hits the cron schedule and calls Hermes gateway.

2
Gateway reads state

Hermes gateway loads jobs.json (job definitions), state.db (parsed job state and cached schedules), and .env (API credentials).

3
Parse cron expression

Gateway parses the job's cron expression (e.g., "0 2 * * *") to determine next run time. If state.db is corrupted, this step fails and shows "?" for schedule and next_run_at.

4
Execute skill script

hermes_run.py invokes the skill script (e.g., daily-sync) with a system prompt and context from .env. Skill runs independently.

5
Capture output

Hermes captures stdout/stderr from the skill script. If the skill succeeded, output is formatted and prepared for delivery.

6
Deliver results

Hermes sends the result to configured channels: Telegram (immediate notification) and Obsidian (persistent log). Job state is updated in state.db.

Root Cause Example: state.db Corruption

Why state.db Issues Block Jobs

✗
Corrupted state.db

If state.db is stale or has malformed job objects, Hermes cannot parse the cron expressions from jobs.json into cached schedules.

✗
Parser fails

Step 3 fails silently or produces incomplete job metadata. next_run_at and schedule display as "?" or "Calculating..."

✗
Job won't execute

Scheduler can't determine if it's time to run the job. Job remains pending and never executes.

✓
Fix: Clean state.db

rm -f ~/.hermes/state.db — Forces Hermes to rebuild scheduler state from clean jobs.json on next restart. Gateway immediately reconstructs valid schedules.
